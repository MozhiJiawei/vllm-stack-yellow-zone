TP4 一轮，不是 4 次或 8 次授权

为什么有 DRAINING / exit_mask

全部 COMPLETE 后，全局 owner 立即释放，对侧可马上获得 grant。本侧 workers 同时进入极短的回收确认：每个 worker 设置 exit bit；全部确认前本实例不能开启下一轮。

这修复了压力测试发现的跨轮竞态：最快 worker 不能在较慢 worker 观察到回收前污染下一轮；又不会阻塞对侧 handoff。

fencing：epoch + instance session + worker session + rank + forward_seq + grant_id。

4. 改动边界

组件当前处理影响
EngineCore / AsyncScheduler不修改不再等待 execute future，不限制 batch_queue，消除旧 P1 sampling 空泡。
vLLM worker 边界UniProc + WorkerProc 两个轻量 patch 点TP1 在模型加载后建单 worker gate；TP≥2 在 MQ 初始化后建位图 gate。两者都只包非空 execute_model，异常 fail-closed。
vLLM-Ascend worker不修改无 stream event、无设备同步、无 READY/End。
vCANN runtime不调用调度接口只保留旧提交前能力；模型间仲裁全部在上层。
控制通道共享 mmap + C11 atomic + futex热路径没有 TCP、UDS、轮询 sleep 或文件 I/O。
性能对照开关VLLM_PAIR_SCHED_DEBUG_BYPASS=1UniProc 与 WorkerProc 都在每个非空 forward 前读取一次；值为 1 时跳过 enter/leave。环境变量读取实测约 0.455 μs/次，仅用于 TTFT/TPOT A/B 测试。

5. 黄区串行启动

A 完全就绪后才启动 B

  1. 启动 A / primary。
  2. /v1/models 成功。
  3. inspector 为 RUNNING。
  4. A registration mask：TP1 = 0b1，TP4 = 0b1111。
  5. 此时才启动 B。

B 完全就绪后才开放业务

  1. /v1/models 成功。
  2. inspector 为 RUNNING。
  3. B registration mask：TP1 = 0b1，TP4 = 0b1111。
  4. 脚本输出 PAIR_READY。

默认超时 900 秒；primary、owner worker 或 barrier 超时均要求 supervisor 整对重启。

6. 当前验证结果

TP1 / TP2 / TP4一次 grant 被本实例全部 worker 消费；全部 COMPLETE 后释放。
零 forward 重叠本地 TP4 双实例 fake WorkerProc 区间检查通过。
sampling 可重叠A sampling 与 B forward 的主机区间实测重叠。
100k rounds零重复 grant、零永久等待；同时发现并修复跨轮竞态。
真实 vLLM 控制流v0.19.1 共 12 个集成用例通过：step_with_batch_queue() 原生轨迹不变，TP1 UniProc 与 TP≥2 WorkerProc 只 gate 非空 forward。
本机性能未过门槛Docker Desktop 固定 CPU 诊断：TP1 P99 324 μs；TP4 P99 1.11 ms。该环境不是黄区裸机，但不能据此宣称性能 PASS。
NPU 未验收集合通信 timeline、设备异步尾部、TPOT 回退与百万轮固定 CPU 指标待黄区。

7. DFX 与交付闸门

立即 FAILED / 整对重启

  • forward deadline 超时或迟到 COMPLETE
  • owner / primary / 执行中 worker 死亡
  • 缺 rank barrier 超时、重复 rank、位图越界
  • epoch、session、sequence 或 grant fencing 不匹配
  • 共享状态损坏

仍需黄区证明

  • 所有 TP worker 主机返回后，没有不可重叠的 NPU 通信尾部
  • A/B 集合通信 timeline 零重叠
  • sampling 与对侧 forward 在设备侧实际重叠
  • 单实例无竞争 TPOT 回退 ≤ 1%
  • 固定 CPU 百万轮 barrier / handoff P99 ≤ 150 μs
